מערכת סבב אישורים

LIVE עודכן: מרץ 2026 גרסה: 1.0

מערכת סבב אישורים מאפשרת ליצור טפסים ייעודיים מסוג approval שבהם כל תגובה עוברת סבב אישורים סדרתי (1..N שלבים). גורמים מאשרים מקבלים קישור מאובטח חד-פעמי ויכולים לאשר או לדחות את הבקשה ללא צורך בהתחברות למערכת.

ℹ️
הפרדה מלאה: הפיצ'ר פועל רק על טפסים מסוג approval. טפסים מסוג general ו-attendance לא מושפעים כלל.

עקרונות מנחים

  • תאימות לאחור מלאה — טפסים קיימים לא מושפעים
  • DB additive בלבד — לא שונתה שום טבלה קיימת
  • אבטחת Token — hash + TTL + single-use
  • יעד מאשר דינאמי — קבוע, משדה, או ממיפוי לפי בחירה
  • שליטה על נראות שדות — בעל הטופס קובע מה המאשר רואה

מודל נתונים

טבלאות חדשות

טבלהתיאור
response_approvalsמופע אישור 1:1 עם response. מחזיק סטטוס כולל, שלב נוכחי, זמני התחלה/סיום.
response_approval_stepsשלב בודד בסבב. מכיל פרטי מאשר, token (hash), סטטוס, חתימה, הערה.
response_approval_eventsAudit log — כל אירוע (אתחול, הנפקת token, החלטה) נרשם כאן.

RPCs (security definer)

פונקציהתיאור
initialize_approval_for_response(p_response_id) פותר יעדים דינאמיים מ-response.data, יוצר מופע + שלבים + token ראשון. רץ רק על טפסי approval.
get_approval_by_token(p_token) מחזיר פרטי בקשה + שדות הטופס + הגדרות נראות — לדף האישור הציבורי.
decide_approval_by_token(p_token, p_decision, ...) מבצע אישור/דחייה, מקדם לשלב הבא או סוגר את הסבב.
✅
אפס שינוי בטבלאות forms, responses, notifications.

זרימת אישור

  1. משתמש ממלא טופס מסוג approval ולוחץ "שלח לאישור".
  2. submitResponse שומר את התגובה ב-responses כרגיל.
  3. מזהה form_type === "approval" → קורא ל-initializeApprovalForResponse.
  4. ה-RPC פותר יעדי מאשרים (קבוע / משדה / ממיפוי), יוצר מופע + שלבים, מנפיק token לשלב הראשון.
  5. Token מוחזר ל-backend → ניתן להעביר למאשר דרך email/whatsapp (adapter חיצוני).
  6. מאשר פותח /approve/{token} → רואה פרטי בקשה (שדות מסוננים) → מאשר / דוחה.
  7. אם אושר ויש עוד שלבים → token חדש לשלב הבא. אם אושר בשלב אחרון → סטטוס approved.
  8. אם נדחה בכל שלב → סטטוס rejected ללא המשך.

יעד מאשר דינאמי

לכל שלב אישור ניתן להגדיר מקור יעד:

מצבשםתיאורדוגמה
fixed יעד קבוע אימייל/טלפון שמוזן ידנית manager@company.com
from_field משדה בטופס הערך שהמשתמש מילא בשדה מסוים (למשל שדה "אימייל מנהל") שדה manager_email → ערך שהוזן
from_option_map ממיפוי לפי בחירה שדה dropdown → מיפוי ערך → יעד יוסי → yossi@co.il
דוד → david@co.il
⚠️
ולידציה: אם ב-runtime הערך שנבחר לא נמצא במיפוי, ה-RPC מחזיר שגיאה missing_target_mapping והתגובה נשמרת אך הסבב לא נוצר (עם warning למשתמש).

נראות שדות למאשר

בעל הטופס קובע אילו שדות המאשר רואה בדף האישור:

מצבתיאור
הכל (mode: "all") כל שדות הקלט מוצגים למאשר — ברירת מחדל
רק מסומנים (mode: "selected") רק שדות עם show_to_approver: true מוצגים. ה-Builder מציג checkbox ליד כל שדה.

הסינון מתבצע בדף /approve/[token] לפי שילוב של approval_field_visibility ו-show_to_approver בכל שדה.

ממשק הגדרה (Builder)

תבנית מהירה

כפתור "תבנית מהירה: סבב אישורים" מחליף את סוג הטופס ל-approval, מגדיר שלב אישור ראשון ריק, ומעדכן את כפתור השליחה ל-"שלח לאישור".

פאנל הגדרות אישורים

מוצג בפאנל ההגדרות רק כאשר formType === "approval":

  • שלבי אישור (1..N) — כל שלב כולל: שם מאשר, ערוץ, מקור יעד, מיפוי
  • נראות שדות — בחירה בין "הכל" ל"רק מסומנים" + checkboxes לשדות
  • Badge — "סבב אישורים" ירוק ב-header

דף אישור ציבורי

נתיב: /approve/[token]

מה המאשר רואה

  • שם הטופס + מספר שלב (X מתוך N)
  • פרטי מאשר (שם, ערוץ, יעד)
  • פרטי הבקשה — רק שדות מותרים, עם Labels מהטופס (לא field IDs)
  • כפתורי "אשר" / "דחה"
  • Textarea להערה + חתימה טקסטואלית (אופציונלי)

מצבים אפשריים

מצבתצוגה
Token תקיןדף אישור מלא עם כפתורי פעולה
Token פג / נוצל"הקישור אינו תקין — נראה שפג תוקף או כבר נוצל"
אחרי אישורהודעה ירוקה: "הבקשה אושרה בהצלחה"
אחרי דחייההודעה אדומה: "הבקשה נדחתה"

לוח אישורים ייעודי

נתיב: /forms/[id]/approvals — זמין רק לטפסים מסוג approval.

עמוד ייעודי (בדומה ללוח נוכחות עבור טפסי attendance) שמציג את כל הבקשות ומצב האישורים שלהן בזמן אמת.

מה כולל העמוד

  • כרטיסי סיכום: סה"כ בקשות, בתהליך, אושרו, נדחו
  • טאבים לסינון: הכל / בתהליך / אושרו / נדחו
  • כרטיס לכל בקשה (מתקפל):
    • סטטוס כללי עם אייקון צבעוני
    • שלב נוכחי (X/N)
    • תקציר שדות מובילים מהטופס
    • שם הגורם המאשר שממתין כרגע (כש-status = in_progress)
  • פירוט מורחב (בלחיצה):
    • ציר זמן ויזואלי של שלבי האישור (timeline)
    • לכל שלב: שם מאשר, ערוץ+יעד, סטטוס, זמן פעולה, הערה
    • פרטי הבקשה המלאים בגריד

תצוגת תוצאות

בדף תוצאות (/forms/[id]/results) של טופס מסוג approval מוצגות שתי עמודות נוספות:

עמודהתיאור

שלב נוכחיX / N (למשל 2/3)

ב-Dashboard, טפסי approval מוצגים עם Badge ירוק "סבב אישורים".

אבטחה

מנגנוןפירוט
Token hashingנשמר רק SHA-256 hash — הערך המקורי לא נשמר ב-DB
TTL72 שעות — אחרי כן Token פג אוטומטית
Single-useאחרי שימוש ראשון token_used_at מתעדכן ו-Token נפסל
Security definerכל RPCs רצים עם security definer — ללא תלות ב-RLS של הקורא
RLSטבלאות approval — קריאה רק לבעל הטופס (join דרך forms.user_id)
Audit logכל פעולה (init, token_issued, step_decision) נרשמת ב-response_approval_events
Minimal exposureדף אישור מציג רק שדות מותרים — לא את כל ה-payload
🔒
חשוב: הקישור /approve/{token} לא מכיל מזהי בקשה, שמות, או מידע מזהה. רק token אקראי בן 48 תווים (hex).
